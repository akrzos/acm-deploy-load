# Result Directory Layout Reference

## acm-deploy-load Results

Directory naming: `{YYYYMMDD-HHMMSS}-{method}-{suffix}`
Example: `20260623-044312-ibi-clusterinstance-gitops-400cpa-500b-1800i-odf-ibi-extended-12`

### Top-Level Files

| File Pattern | Description | Format |
|---|---|---|
| `report.txt` | Test summary report card | Plaintext (see format below) |
| `deploy-time-{TS}` | Install/deployment milestones | Plaintext (see format below) |
| `monitor_data.csv` | Time-series workload state, 1-min intervals | CSV |
| `imageclusterinstalls-{TS}.stats` | IBI per-cluster install timing | Plaintext stats |
| `imageclusterinstalls-{TS}.csv` | IBI per-cluster timing detail | CSV |
| `imageclusterinstalls-{TS}.json` | IBI per-cluster raw data | JSON |
| `clusterinstances-{TS}.stats` | ClusterInstance provisioning timing | Plaintext stats |
| `clusterinstances-{TS}.csv` | ClusterInstance timing detail | CSV |
| `clusterinstances-{TS}.json` | ClusterInstance raw data | JSON |
| `clustergroupupgrades-ztp-install-{TS}.stats` | CGU per-cluster policy timing | Plaintext stats |
| `clustergroupupgrades-ztp-install-{TS}.csv` | CGU timing detail | CSV |
| `agentclusterinstalls-{TS}.stats` | AI per-cluster install timing (AI methods only) | Plaintext stats |
| `agentclusterinstalls-{TS}.csv` | AI per-cluster timing detail | CSV |
| `cluster-{TS}.png` | Cluster state graph | PNG |
| `policy-{TS}.png` | Policy state graph | PNG |
| `share-{TS}.png`, `share2-{TS}.png` | Share graphs | PNG |
| `must-gather-{TS}.tar.gz` | OpenShift must-gather archive | tar.gz |
| `iz-all-{TS}.log.gz` | Compressed install logs | gzip |
| `pa-{phase-name}.log` | Prometheus analysis stdout/stderr per phase | Log |

### Phase Directories (Prometheus Analysis per Test Phase)

Present when start-delay or end-delay >= 15 minutes.

**Current naming** (prefix includes phase number):
- `phase1-idle-baseline-{YYYYMMDD-HHMMSS}/`
- `phase2-cluster-deployment-{YYYYMMDD-HHMMSS}/`
- `phase3-soak-baseline-{YYYYMMDD-HHMMSS}/`

**Older naming** (no phase number prefix):
- `idle-baseline-{YYYYMMDD-HHMMSS}/`
- `cluster-deployment-{YYYYMMDD-HHMMSS}/`
- `soak-baseline-{YYYYMMDD-HHMMSS}/`

### Full-Test Prometheus Analysis Directories (Orchestration Script)

These directories are generated by the orchestration scripts (not by the Python
tools directly) and cover the entire test duration (start to end time from
report.txt):

- acm-deploy-load: `deploy-pa-{YYYYMMDD-HHMMSS}/` — generated by
  `scripts/interval-ztp-install-all.sh` with `-p deploy-pa`
- acm-telco-core-load: `acm-telco-load-hub-{YYYYMMDD-HHMMSS}/` — generated by
  `scripts/acm-telco-core-load.sh` with `-p acm-telco-load-hub`
- `deploy-pa-all-{YYYYMMDD-HHMMSS}/` — generated ad-hoc manually, ignore unless
  user specifically requests

### Prometheus Analysis Directory Internal Structure

Each phase directory and deploy-pa-* directory has the same internal layout:

```
{analysis-dir}/
├── analysis                    # Metadata: cluster version, time range, duration
├── report.html                 # HTML report with embedded graph links
├── cluster/
│   ├── {metric}.png
│   ├── csv/{metric}.csv
│   └── stats/{metric}.stats
├── node/
│   ├── {metric}.png
│   ├── csv/{metric}.csv
│   └── stats/{metric}.stats
├── etcd/
│   ├── csv/{metric}.csv
│   └── stats/{metric}.stats
├── resource/
│   ├── csv/{metric}.csv
│   └── stats/{metric}.stats
├── base-ocp/
│   └── stats/{metric}.stats
├── core-ocp/
│   └── stats/{metric}.stats
├── {component}/                # One per detected component
│   ├── {metric}.png
│   ├── csv/{metric}.csv
│   └── stats/{metric}.stats
```

**Stats files are always under the `stats/` subdirectory** — never directly
under the component directory.

### install-data/ Directory

```
install-data/
├── clusterinstance.yaml, aci.yaml, ici.yaml
├── cgu.status, .yaml
├── nodes, pods, namespaces, events
├── storageclasses, .yaml          # StorageClasses, PVs, PVCs
├── pv, pvc
├── odf.*                          # ODF data (only if ODF installed)
├── policy, policy.yaml
├── cgu-failures/                  # Per-cluster failure details
└── vm{NNNNN}-clusterinstance.yml  # Per-cluster configs
```

## acm-telco-core-load Results

Directory naming: `{YYYYMMDD-HHMMSS}-telco-core-load-{suffix}`

Same structure as acm-deploy-load with these additions/differences:
- `policy-cm/` directory containing policy ConfigMap YAML snapshots
- Phase 2 may produce batch sub-phases: `phase2-batch-{N}-{YYYYMMDD-HHMMSS}/`
- No ICI/ACI stats files (telco-core uses ClusterInstance GitOps only)
- Report card title: `acm-telco-core-load Report Card`

## Workload Timing Stats File Format

The `.stats` files at the result top level (ICI, CGU, CI, ACI) use a **plaintext
format**, NOT the pandas describe format used by Prometheus stats. Example:

```
Stats only on ImageClusterInstalls CRs in ClusterInstallationSucceeded
Count: 324
Min: 390.0
Average: 675.0
50 percentile: 684.0
95 percentile: 749.0
99 percentile: 802.5
Max: 1083.0
```

The CGU stats file has additional header lines:

```
#############################################
Stats on clustergroupupgrades CRs in namespace ztp-install
Total CGUs - 324
Skipped local-cluster CGU
CGUs with Completed: 324 - 100.0%
Earliest CGU creationTimestamp: 2026-06-23 05:55:13

Duration between creation and startedAt: 0.0s :: 0:00:00
Duration between startedAt and completedAt: 995.0s :: 0:16:35

Count: 324

95 percentile: 390.8

```

The ClusterInstance stats file contains two stat blocks: one for CreationTimestamp
until InstanceValidated, and one for Total Duration (Provisioned):


Use the "Total Duration" block for the ClusterInstance end-to-end provisioning time.

## Prometheus Stats File Format (pandas describe)

Stats files under `{component}/stats/` use pandas `describe()` output:

```
       d37-h14-000-r660  d38-h11-000-r660  d38-h12-000-r660
count         60.000000         60.000000         60.000000
mean           3.118071          4.234567          3.987654
std            0.704942          0.823456          0.654321
min            2.112000          3.123456          2.987654
25%            2.697083          3.654321          3.456789
50%            2.967833          4.123456          3.876543
75%            3.372417          4.567890          4.234567
95%            4.263050          5.654321          5.123456
99%            5.332863          6.234567          5.876543
max           12.535333          7.654321          6.234567
```

- Column headers are node names, pod names, namespace names, or aggregate labels
- Rows: count, mean, std, min, 25%, 50%, 75%, 95%, 99%, max
- Single-column files use a label like `cluster` or `[fd01:0:0:2::65]:9283`
